Phần 3 · Công cụ

Nền: Claude Code — Lớp điều phối: oh-my-claudecode — Điểm khác: oh-my-pi

Claude Code (nền)

CLI agent terminal-native (preview 10/2024); "do the simple thing first" — vòng lặp đơn luồng đọc→sửa→chạy→tự kiểm, bạn duyệt kết quả. Mạnh, minh bạch — nhưng trần năng suất nằm ở chỗ là một agent.

oh-my-pi (điểm khác)

Harness khác, cùng triết lý delegation-first, nhấn vào bộ tool tích hợp + song song hóa (AST/LSP/eval/browser/IRC). OMC dễ tiếp cận hơn cho người mới.

oh-my-claudecode — lớp điều phối đa-agent (chồng lên, không thay thế)

  • 19 agent chuyên biệt theo lane (exploration/planning/execution/verification/review/testing…), gắn model tier hợp lý (Haiku/Sonnet/Opus)
  • Execution modes: Autopilot, Team, Ralph, Ultrawork, UltraQA
  • MCP tools: LSP, AST grep, Python REPL, state/notepad/project-memory, model ngoài (Codex, Gemini)
Golden Rule: "NEVER make code changes directly. ALWAYS delegate to specialized agents." — vai trò bạn là guide/review/orchestrate

Nguồn: oh-my-claudecode docs (#execution-modes) · anthropic.com/product/claude-code

Phần 4 · Hệ thống level

Thang 6 bậc tự chủ — grounded trên Swarmia + CSA + Knight + SAE J3016 analogy

Tương tự SAE J3016 (6 cấp tự lái xe); hợp nhất từ 4 khung độc lập (Swarmia coding-specific, CSA human-control, Knight vai trò người dùng, ELEKS maturity) — đồng thuận: 5–6 bậc, cao hơn KHÔNG đồng nghĩa tốt hơn.

Phần 4 · Hệ thống level

Cao hơn không tốt hơn — guardrail mới là điều kiện lên bậc

  • Chỉ nên leo từ Ln lên Ln+1 khi đã có: (1) test tự động bao quanh vùng agent đụng tới, (2) quy trình review, (3) spec/acceptance rõ
  • Thiếu guardrail → bậc cao hơn chỉ phóng đại rủi ro (bằng chứng ở slide sau)
  • L4 orchestration rực rỡ cho việc song song, lặp lại, có test bao quanh; L2 supervised đúng cho code nhạy cảm, ít test, một-lần
L0 — L1 — L2 · guardrail đủ → mở khóa L3+

Phần 4 · Hệ thống level

AI hiệu quả khi đúng chỗ — đo bằng metric hệ thống, không phải cảm giác nhanh

Mặt tích cực

  • Copilot lab: nhanh hơn 55.8% trên task đơn giản — giảm còn ~42% ở real-world (Peng 2023)
  • Junior +21–52%, senior +7–16%; task càng phức tạp lợi ích càng tiệm cận <10% (McKinsey)

Metric đáng tin (DORA/SPACE)

  • Code churn rate
  • Change Failure Rate
  • Lead time for changes
  • Defect escape rate
  • Security vuln density
Mỗi bậc autonomy là một công cụ. Leo lên L4/L5 mà thiếu guardrail = đúng công thức tạo churn và lỗ hổng mà nghiên cứu trên đo được.

Nguồn: arXiv 2302.06590 (Peng 2023) · McKinsey

Phần 5 · Cảnh 1

deep-interview: hỏi "bạn đang ngầm giả định gì", không hỏi "bạn muốn gì"

deep interview: thêm đăng nhập Google vào API
  • Một câu 10 từ — giao thẳng cho AI, nó sẽ đoán: session hay JWT? refresh token? user trùng email? Mỗi cú đoán sai = 1 vòng rework
  • Chốt hình dạng việc trước (Round 0): "3 mảnh độc lập — luồng OAuth, lưu/khớp user, phát token. Đúng chưa?"
  • Mỗi vòng sau nhắm mảnh mờ nhất, hỏi đúng 1 câu — ambiguity giảm dần, dưới ngưỡng (mặc định 20%) → kết tinh spec vào .omc/specs/
59%→42%→28%→15%
Bug chưa rõ nguyên nhân → /deep-dive (truy vết song song, tự nạp phát hiện vào phiên deep-interview).

Phần 5 · Cảnh 2

ralplan: một hội đồng 3 người tranh luận cách làm

ralplan
  • Planner dựng bước → Architect soi ranh giới hệ thống, đánh đổi dài hạn → Critic đập lại ("user đã có tài khoản local trùng email thì sao?", "OAuth callback đã chống CSRF chưa?")
  • Lặp tối đa 5 lần; đầu ra: plan có Decision Drivers + ≥2 phương án (ADR) → .omc/plans/
Plan dừng ở pending approval — KHÔNG tự chạy tiếp. Bạn đọc kế hoạch, không đọc từng dòng code, rồi mới quyết.

Cổng đồng thuận: rõ ràng (deep-interview) → khả thi (ralplan) → cho phép thực thi (bạn)

Gõ thẳng "ralph thêm OAuth" cụt lủn → ralplan vẫn chặn (thiếu "neo" cụ thể), ép qua vòng consensus. Bỏ qua cố ý → tiền tố force:

Phần 5 · Cảnh 4

Cơ chế dừng-an-toàn khi có sự cố

Tình huống: Ralph quay vòng — cùng lỗi test OAuth callback hỏng 3 lần liên tiếp.

  • Ralph không lăn vô hạn mù quáng — gặp cùng lỗi 3 vòng → escalate thay vì cố mãi
  • Tín hiệu "The boulder never stops" trong log = vòng lặp đang chạy bình thường, không phải bị treo
/trace     # xem timeline luồng agent — kẹt ở đâu
/oh-my-claudecode:cancel --force   # reset session + state cũ

Mẹo gỡ kẹt nhanh: "not inside tmux" → tmux new -s dev · "codex: command not found" → cài CLI ngoài · hook lạ → DISABLE_OMC=1 hoặc OMC_SKIP_HOOKS=<tên>

Phần 5 · Cảnh 5

Validation: 3 người khác phải đồng ý, không phải người viết

  • Ralph báo mọi story passes, Architect đã duyệt trong Execution — nhưng autopilot chưa cho qua
  • 3 reviewer bắt buộc: Architect (kiến trúc) · Security-reviewer (CSRF/secret/token rò rỉ) · Code-reviewer (chất lượng tổng thể)
  • Golden Rule đóng thành quy trình: người viết code không được tự duyệt
  • Cleanup bắt buộc: ai-slop-cleaner quét bỏ phần phình → chạy lại test

✓ Architect   ✓ Security-reviewer   ✓ Code-reviewer → merge

Nghiệm thu bằng bằng chứng (test pass, churn/CFR/lead time) — không bằng cảm tính. OAuth lên PR; merge.

Phần 5 · Cảnh 6

Vì sao lần sau nó không hỏi lại

Tuần sau, thêm "đăng nhập GitHub". Agent không hỏi lại tech stack, quy ước đặt tên, hay việc dự án dùng JWT.

  • Quyết định dài hạn → .omc/project-memory.json
  • Ghi chú cao-signal → .omc/notepad.md (<remember> = 7 ngày, <remember priority> = vĩnh viễn)
  • Plan/spec cũ → .omc/plans/, .omc/specs/
  • Bài học OAuth callback → đã "wiki this" → tri thức tra cứu được cho mọi phiên sau

"Dàn nhạc nhớ bản tổng phổ; bạn không phải dạy lại từ đầu."
